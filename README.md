## ファイルを保存してプッシュする

- プッシュ：手元のセーブデータを、共有のサーバーに反映させること。

```jsx
#すべての変更を保存対象(準備)
git add .

#特定のファイル(Ex introduction.py, .gitignore)
git add introduction.py .gitignore

#セーブ（コミット）する(記録)
git commit -m "メッセージ"

#GitHubに送る(共有)
#最初だけ「-u origin main」と書くことで、次から「git push」だけで済む
git push -u origin main

#注意
#「プッシュしたつもりが、コミットを忘れていた」を防ぐ
git status
```

## メッセージの作法

`feat: Python基礎コードの追加` のように、**[種別]: [内容]** という形式（Conventional Commitsと言う）で書く

| **種別 (Type)** | **内容** | **具体的な使用例** |
| --- | --- | --- |
| **`feat`** | **新機能の追加** (feature) | `feat: ログイン機能の実装` |
| **`fix`** | **バグの修正** | `fix: 入力フォームのバリデーションエラーを修正` |
| **`docs`** | **ドキュメントのみの修正** (documentation) | `docs: READMEに使用方法を追記` |
| **`style`** | **見た目の調整**（コードの動作に影響しない、空白やセミコロンなど） | `style: インデントを4スペースに統一` |
| **`refactor`** | **リファクタリング**（バグ修正でも新機能でもないコード整理） | `refactor: 重複していた計算処理を共通関数に統合` |
| **`test`** | **テストコードの追加・修正** | `test: ユーザー登録のユニットテストを追加` |
| **`chore`** | **雑用・ビルド構成の変更**（ソースコード以外、ライブラリの更新など） | `chore: termcolorライブラリを追加` |

### 1. Gitとは何か？どのようなツールか？

- **Gitの基本的な役割**: ファイルの変更履歴を時系列で保存し、管理するための「分散型バージョン管理システム」。
- **できること**:
    - **変更の記録**: 「いつ、誰が、何を」変えたかをセーブポイント（コミット）として残す。
    - **ブランチ作成**: 本番用の道から枝分かれして、安全に新機能の開発を行う。
    - **マージ**: 枝分かれした作業を、本番の道へ合流させる。

### 2. Gitはどのように「分散型」なのか？

- **分散型バージョン管理システム (DVCS) とは**: 中央サーバーだけでなく、開発者全員が自分のPCに「履歴の全コピー」を持つ仕組み。
- **集中型（SVNなど）との違い**:
    - **集中型**: 常にサーバーに接続していないと履歴が見られない。サーバーが壊れると全履歴が消えるリスクがある。
    - **分散型**: ネットがなくても自分のPCで履歴確認やコミットが可能。全員がバックアップを持っている状態なので壊れにくい。
- **ローカルで履歴管理できる利点**: 動作が非常に高速。試行錯誤の過程を細かく「自分だけのセーブデータ」として残し、完成してから共有できる。

### 3. コンフリクト（衝突）

Gitがどちらの変更を優先すればいいか判断できない場合に生じる。主に「同じファイルの、同じ行」を、2人（または2つのブランチ）が別々に書き換えて、それを合流（マージ）させようとした時に発生。

- コンフリクトの解消手順（VS Code編）
    - **VS Codeでファイルを開く**: 該当箇所がハイライトされ、上に小さな文字で選択肢が出る。
        - **Accept Current Change**: 自分の変更を採用する。
        - **Accept Incoming Change**: 相手の変更を採用する。
        - **Accept Both Changes**: 両方の変更を並べて残す。
    - **手動で修正**: 記号（`<<<<`など）を全部消して、最終的に残したい綺麗なコードに書き換えます。
    - **解決を報告**:
    
    ```
    git add <解消したファイル名>
    git commit -m "fix: コンフリクトを解消"
    ```
    
- コンフリクトを防ぐための「お作法」
    - **こまめに `pull` する**: リモートの最新状態を常に手元に取り込んでおけば、差分が小さいうちに解決。
    - **ブランチの寿命を短くする**: 何日も放置したブランチはコンフリクトしやすくなる。早めにマージ。
    - **作業範囲を分ける**: 同じファイルを複数人で同時にいじらないよう、タスクを分担。

### 4.  リベース **(Rebase)**

ブランチの付け根（ベース）を、最新の状態に移動させる」操作

現在の開発現場では、`git pull` よりも `git fetch` を使って「何が変わったかを確認してから取り込む」流れが推奨

```bash
# 1. リモートの最新情報を取得
git fetch origin

# 2. 自分のブランチの根っこを、最新の main の上に「植え替える」
git rebase origin/main

# 3. 【もしコンフリクトしたら】
# VS Codeなどで修正し、準備ができたら add
git add .
# 修正終完了、次のコミットに進める
git rebase --continue

# 4. 【もし途中でパニックになったら】
# リベースなんて無かったことに
git rebase --abort

# 5. GitHubへ反映（歴史を書き換えたので、普通のpushは拒否される）
git push --force-with-lease
```

### `-force` よりも `-force-with-lease` を使う理由

- **`-force` (危険)**: GitHub側の内容を問答無用で上書き。もし自分が作業している間に「他の誰か」が同じブランチにプッシュしていても、それを消し去る。
- **`-force-with-lease` (安全)**: 「自分が最後に見た状態から、GitHub側が更新されていなければ」上書き。**もし他の人がプッシュしていたら、エラーで止まってくれる**ため、仲間を事故に巻き込まない。
- リベースはあくまで **「自分の手元の作業を、本番に合流させる直前に整えるため」** に使う。

### 5. ローカルリポジトリとリモートリポジトリの違い

- **各リポジトリの役割**:
    - **ローカル**: 自分のPC内の作業場兼、自分専用の履歴置き場。
    - **リモート**: GitHubなどのサーバー上にある、チーム全員で共有する履歴置き場。
- **ローカルの操作**: `add`（買い物カゴに入れる）、`commit`（レジで会計＝履歴に記録）。
- **リモートとの同期**: `push`（自分の履歴を共有場所に送る）、`pull`（他人の履歴を手元に取り込む）。
- リモートリポジトリのクローン

```jsx

##クローン実行

```

### 6. Gitの主な利点（具体的開発場面）

- **変更履歴の管理（具体例）**:
    1. **「昨日まで動いていたのに…」**: 昨日のコミットと今日のコードの差分を比較して、バグを1分で特定する。
    2. **「仕様変更で元に戻したい」**: 3日前の状態に一瞬でタイムトラベルする。
    3. **「証跡の確認」**: 1年前のコードを見て「なぜこの複雑な処理を書いたのか」を当時のメッセージから読み解く。
- **協調作業のサポート（具体例）**:
    1. **並行開発**: Aさんが「ログイン機能」、Bさんが「検索機能」を、お互いの作業を壊さずに同時に開発する。
    2. **コードレビュー**: `push` されたコードをマージする前に、リーダーが内容をチェックして指摘を入れる。

### 7. 他のシステムとの比較・なぜGitが主流か

- **比較**:
    - **SVN**: 構造が単純だが、ブランチ作成やマージが重く、大規模開発に向かない。
    - **Mercurial**: Gitに近いが、Gitほど自由度が高くなかったため普及で一歩遅れた。
- **なぜGitが主流か**:
    1. **圧倒的な速度**: ほとんどの操作がローカルで完結するため爆速。
    2. **柔軟なブランチ運用**: 枝分かれと合流が非常に簡単で低コスト。
    3. **エコシステム**: GitHubという最強のプラットフォームや、VS Codeなどのツール連携が強力。

- 「Public」と「Private」の違い

| **項目** | **Public (公開)** | **Private (非公開)** |
| --- | --- | --- |
| **閲覧権限** | 世界中の誰でも閲覧・クローンが可能 | 自分と許可したメンバーのみ |
| **実績（草）** | プロフィールの活動グラフに反映される | 以前は反映されなかったが、現在は設定で反映可能 |
| **主な用途** | 学習成果、OSS、ポートフォリオ | **業務コード**、機密情報、練習用リポジトリ |
| **今回の運用** | 研修課題（athena-tech-onboarding）に使用 | 練習用（python-template）に使用 |
- リポジトリの命名規則（使える文字、命名のベストプラクティス）（Naming Convention）
    - **使える文字**: 英小文字、数字、ハイフン ()、アンダースコア (`_`)、ピリオド (`.`)。
    - **ベストプラクティス**:
        - **ケバブケース (`kebab-case`)**: `python-training-task` のように、小文字とハイフンで繋ぐのが最も一般的。
        - **簡潔さ**: `my-python-programming-study-repository` よりも `python-study` のように、一目で内容がわかる名前を心がける。
        - **避けるべきもの**: スペース、日本語、大文字の混在。
- HTTPSとSSHの違い（どちらを使うべきか？）
    
    
    | **項目** | **HTTPS（現在の設定）** | **SSH** |
    | --- | --- | --- |
    | **URL形式** | `https://github.com/hoso-jpn/...` | `git@github.com:hoso-jpn/...` |
    | **認証の仕組み** | **トークンベース**
    GitHub CLI (`gh`) が代行 | **鍵ペアベース**
    公開鍵と秘密鍵を使用 |
    | **設定の難易度** | **非常に簡単**
    ブラウザログインで完結 | **中程度**
    鍵の生成・登録が必要 |
    | **セキュリティ** | **非常に高い**
    OSの保管庫 (keyring) で保護 | **最強レベル**
    物理的な鍵ファイルで認証 |
    | **ネットワーク** | 制限が少ない（どこでも繋がる） | 厳しい会社の回線では遮断されることも |
    
    HTTPS (Hypertext Transfer Protocol Secure)
    
    - **メリット**:
        - **GitHub CLI (`gh`) との相性がよい**: Privateリポジトリでもパスワード入力なしで操作可能。
    - デメリット
        - `gh` などのツールがない環境では、毎回「パーソナルアクセストークン」を入力する必要があり、管理が面倒。
    
    SSH (Secure Shell)
    
    - **メリット**:
        - **一度設定すれば永続的**: 鍵ファイルさえPCにあれば、ブラウザログインすら不要。
        - **CI/CDとの相性**: サーバーからGitHubへ自動アクセスさせる際など、人間が介在しない自動化処理でよく使用。
        - CI/CDパイプラインとSSHと相性が良い。
    - **デメリット**:
        - **鍵管理の責任**: 「秘密鍵」を紛失したり盗まれたりすると、他人が成り代わってアクセスできる。
    
    ## CI/CDパイプラインとは
    
    ### CI：継続的インテグレーション (Continuous Integration)
    
    「インテグレーション（統合）」を「継続的」に行うという意味。
    
    - **何をするか**: 開発者がコードを GitHub などにプッシュ（統合）するたびに、**自動でビルドやテストを実行**。
    - **メリット**: 誰かが書いたコードが既存の機能を壊していないか（バグがないか）を即座に判定。
    - **現場の感覚**: 「誰かのミスでアプリが動かなくなった！」という事態を、リリース前に機械が未然に防いでくれる。
    
    ### 2. CD：継続的デリバリー / デプロイ (Continuous Delivery / Deployment)
    
    テストに合格したコードを、**自動で本番環境（サーバーなど）に届ける**仕組み。
    
    - **継続的デリバリー**: 本番へ出す直前の状態まで自動で準備し、最後の「リリースボタン」だけ人間が押す形式。
    - **継続的デプロイ**: テストを通過したら、**人間を介さずそのまま本番公開**まで自動で行う形式。
    - **メリット**: 手動でファイルをアップロードする際の手順ミス（先祖返りやファイル漏れ）がゼロ。
    
    ## SSHの設定
    
    - 自分のPCで `ssh-keygen` を実行して鍵ペアを作る。
    - `id_rsa.pub`（公開鍵）の中身をGitHubの「SSH and GPG keys」に登録する。
    - URLを `git@github.com:hoso-jpn/...` に切り替える。
- クローン後のディレクトリ構造を確認する方法（`tree` や `ls` コマンドなど）
    
    ### 1. `ls` コマンド（基本の確認）
    
    「List」の略で、現在のディレクトリにあるファイルやフォルダを表示。
    
    - **`ls`**: 通常の表示。
    - **`ls -a` (All)**: **`.git` フォルダ**などの隠しファイルも含めて全て表示。
        
        > **💡 ポイント**: クローン直後のリポジトリで `.git` が見えれば、Git管理が正常に始まっている証拠。
        > 
    - **`ls -la` (Long + All)**: ファイルの権限、所有者、サイズ、更新日時を詳しく表示。
    
    ### 2. `tree` コマンド（構造の可視化）
    
    ディレクトリを樹形図のように表示し、全体の階層構造を一目で把握。
    
    - **`tree`**: 全ての階層を表示。
    - **`tree -L 2` (Level)**: 表示する深さを指定（この場合は2階層目まで）。プロジェクトが巨大な時に便利。
    - **`tree -I ".venv"` (Ignore)**: 特定のフォルダを除外して表示。
        
        > **💡 ポイント**: Pythonプロジェクトでは `.venv` の中に膨大なライブラリが入っているため、`-I ".venv"` を使わないと画面が埋まってしまうことがある。
        > 
    
    ### 3. `git status`（Git視点の確認）
    
    「Gitが現在の状態をどう認識しているか」を確認する最重要コマンド。
    
    - クローン直後に実行して **"nothing to commit, working tree clean"** と出れば、GitHubと手元の状態が完全に一致している。
