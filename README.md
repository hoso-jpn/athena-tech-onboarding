## 研修1：Git の基本操作

### Git を使い始める（初期化）

```bash
git init
```

---

### GitHub のリポジトリと紐付ける

```bash
git remote add origin https://github.com/hoso-jpn/athena-tech-onboarding.git
```

---

### .gitignore を作成する

```bash
echo ".venv/" > .gitignore
```

> **`.gitignore` とは**：仮想環境（`.venv`）、OS 固有のキャッシュ、機密情報（API キーなど）を誤って共有しないようにするための設定ファイル。

---

### ファイルを保存してプッシュする

> **プッシュ**：手元のセーブデータを、共有サーバーに反映させること。

```bash
# すべての変更をステージング（準備）
git add .

# 特定のファイルのみステージング（例：introduction.py と .gitignore）
git add introduction.py .gitignore

# コミット（変更を記録）
git commit -m "コミットメッセージ"

# GitHub に送信（プッシュ）
# 初回のみ `-u origin main` を指定することで、次回から `git push` のみで OK
git push -u origin main

# 現在の状態を確認（「コミット忘れたままプッシュ」を防ぐ）
git status
```

---

### コミットメッセージの書き方

`feat: Python 基礎コードの追加` のように、**`[種別]: [内容]`** の形式（Conventional Commits）で記述します。

| 種別 | 内容 | 使用例 |
|---|---|---|
| `feat` | 新機能の追加（feature） | `feat: ログイン機能の実装` |
| `fix` | バグの修正 | `fix: 入力フォームのバリデーションエラーを修正` |
| `docs` | ドキュメントのみの修正 | `docs: README に使用方法を追記` |
| `style` | 見た目の調整（動作に影響しない変更） | `style: インデントを 4 スペースに統一` |
| `refactor` | リファクタリング（バグ修正・新機能ではないコード整理） | `refactor: 重複していた計算処理を共通関数に統合` |
| `test` | テストコードの追加・修正 | `test: ユーザー登録のユニットテストを追加` |
| `chore` | 雑務・ビルド設定の変更（ライブラリの更新など） | `chore: termcolor ライブラリを追加` |

### Git に関する重要概念

#### 1. Git とは

- **役割**：ファイルの変更履歴を時系列で保存・管理する「分散型バージョン管理システム」。
- **主な機能**：
  - **変更の記録**：「いつ・誰が・何を」変えたかをコミットとして残す。
  - **ブランチ作成**：本番環境から切り離した安全な作業空間を作る。
  - **マージ**：ブランチでの変更を本番ブランチに合流させる。

#### 2. 「分散型」とは

- **分散型バージョン管理システム（DVCS）**：中央サーバーだけでなく、開発者全員が自分の PC に「履歴の全コピー」を持つ仕組み。

| 比較項目 | 集中型（SVN など） | 分散型（Git） |
|---|---|---|
| **オフライン時** | 履歴の確認・操作が不可 | ローカルで履歴確認・コミットが可能 |
| **サーバー障害時** | 全履歴が失われるリスクあり | 全員がバックアップを保持しているため安全 |
| **速度** | ネットワーク依存 | ほとんどの操作がローカル完結で高速 |

#### 3. コンフリクト（衝突）

「同じファイルの同じ行」を複数人が別々に変更し、マージしようとした際に発生します。Git がどちらの変更を優先すべきか判断できない状態です。

**VS Code での解消手順**

1. 該当ファイルを開く（衝突箇所がハイライトされる）。
2. 表示される選択肢から対応を選ぶ：
   - **Accept Current Change**：自分の変更を採用。
   - **Accept Incoming Change**：相手の変更を採用。
   - **Accept Both Changes**：両方の変更を残す。
   - **手動修正**：`<<<<` などの記号を削除し、最終的なコードに直接書き換える。
3. 解消後に報告：

```bash
git add <解消したファイル名>
git commit -m "fix: コンフリクトを解消"
```

**コンフリクトを防ぐために**

- **こまめに `pull` する**：リモートの最新状態を常に取り込み、差分を小さく保つ。
- **ブランチの寿命を短くする**：長期間放置したブランチはコンフリクトしやすいため、早めにマージする。
- **作業範囲を分ける**：同じファイルを複数人が同時に編集しないよう、タスクを分担する。

#### 4. リベース（Rebase）

「ブランチの付け根（ベース）を最新の状態に移動させる」操作です。

現場では `git pull` よりも `git fetch` で変更内容を確認してから取り込む流れが推奨されています。

```bash
# 1. リモートの最新情報を取得
git fetch origin

# 2. 自分のブランチの根っこを最新の main に「植え替える」
git rebase origin/main

# 3. コンフリクトが発生した場合
#    VS Code などで修正後、ステージングする
git add .
#    修正完了、次のコミットへ進める
git rebase --continue

# 4. リベースを中断して元の状態に戻す場合
git rebase --abort

# 5. GitHub へ反映（履歴の書き換えを伴うため --force-with-lease を使用）
git push --force-with-lease
```

**`--force` より `--force-with-lease` を使う理由**

| コマンド | 挙動 |
|---|---|
| `--force` | GitHub 側の内容を問答無用で上書き。他の人のプッシュも消去してしまう危険がある。 |
| `--force-with-lease` | 「自分が最後に確認した状態から GitHub が更新されていなければ」上書き。他者のプッシュが検出された場合はエラーで止まる。 |

> **注意**：リベースはあくまで「自分の作業を本番ブランチに合流させる直前に整えるため」に使います。

### 参考：SourceTree（GUI ツール）

「全体の流れやログの確認は UI で、操作は慣れたコマンドで」という使い分けができます。

**インストール手順**

3. GitHub と連携する場合：設定 → アカウント → 追加 → ホスト: GitHub / プロトコル: HTTPS → 「アカウントを接続」

**ローカルリポジトリの追加**

「ローカル」タブ → 「ディレクトリをスキャン」でクローン済みリポジトリを自動検出できます。

### ローカルリポジトリとリモートリポジトリの違い

| 種類 | 役割 | 主な操作 |
|---|---|---|
| **ローカル** | 自分の PC 内の作業場・個人用履歴置き場 | `add`（ステージング）、`commit`（記録） |
| **リモート** | GitHub などのサーバー上にある共有履歴置き場 | `push`（送信）、`pull`（取り込み） |

**リモートリポジトリのクローン**

```bash
# 作業用ディレクトリに移動
cd ~/projects

# クローン実行
git clone <コピーした URL>
```

---

### Git の主な利点

**変更履歴の管理**

1. 「昨日まで動いていたのに…」→ 昨日のコミットとの差分を比較してバグを特定。
2. 「仕様変更で元に戻したい」→ 過去の状態に素早く復元。
3. 「証跡の確認」→ 古いコミットメッセージから当時の意図を読み解く。

**協調作業のサポート**

1. **並行開発**：機能ごとにブランチを分けることで、互いの作業を干渉させずに開発できる。
2. **コードレビュー**：プッシュされたコードをマージ前にリーダーが確認・指摘できる。

---

### 他のシステムとの比較

| ツール | 特徴 |
|---|---|
| **SVN** | 構造がシンプルだが、ブランチ作成やマージが重く大規模開発には不向き。 |
| **Mercurial** | Git に近い設計だが、柔軟性の差で普及で一歩遅れた。 |
| **Git** | 高速・柔軟なブランチ運用・GitHub エコシステムとの連携が強力で業界標準。 |

## 研修2：GitHub の設定と基礎知識

### Public と Private の違い

| 項目 | Public（公開） | Private（非公開） |
|---|---|---|
| **閲覧権限** | 誰でも閲覧・クローン可能 | 自分と許可したメンバーのみ |
| **活動グラフへの反映** | 反映される | 設定により反映可能 |
| **主な用途** | 学習成果・OSS・ポートフォリオ | 業務コード・機密情報・練習用 |
| **今回の運用** | 研修課題（athena-tech-onboarding） | 練習用（python-template） |

---

### リポジトリの命名規則

- **使える文字**：英小文字・数字・ハイフン（`-`）・アンダースコア（`_`）・ピリオド（`.`）
- **推奨形式**：ケバブケース（例：`python-training-task`）
- **避けるべきもの**：スペース・日本語・大文字の混在

---

### HTTPS と SSH の違い

| 項目 | HTTPS | SSH |
|---|---|---|
| **URL 形式** | `https://github.com/hoso-jpn/...` | `git@github.com:hoso-jpn/...` |
| **認証方式** | トークンベース（GitHub CLI が代行） | 鍵ペアベース（公開鍵・秘密鍵） |
| **設定の難易度** | 非常に簡単（ブラウザログインで完結） | 中程度（鍵の生成・登録が必要） |
| **セキュリティ** | 非常に高い（OS の保管庫で保護） | 最強レベル（物理的な鍵ファイルで認証） |
| **ネットワーク制限** | 制限が少ない | 厳しい企業回線では遮断されることがある |

**HTTPS のメリット・デメリット**

- ✅ GitHub CLI（`gh`）との相性が良く、Private リポジトリでもパスワードなしで操作可能。
- ❌ `gh` がない環境では Personal Access Token の入力が都度必要で管理が煩雑。

**SSH のメリット・デメリット**

- ✅ 一度設定すれば永続的に使用可能。CI/CD パイプラインでの自動化処理とも相性が良い。
- ❌ 秘密鍵の紛失・盗難時に不正アクセスのリスクがある。

---

### CI/CD パイプラインとは

#### CI（継続的インテグレーション）

コードをプッシュするたびに**自動でビルド・テストを実行**し、既存機能への影響を即座に検出する仕組み。

#### CD（継続的デリバリー / デプロイ）

テストを通過したコードを**自動で本番環境に配信**する仕組み。

| 種類 | 内容 |
|---|---|
| **継続的デリバリー** | 本番リリース直前まで自動化。最後の「リリースボタン」だけ人間が押す。 |
| **継続的デプロイ** | テスト通過後、人間を介さず本番公開まで全自動で実行。 |

---

### SSH の設定手順

1. ターミナルで鍵ペアを生成する：

```bash
ssh-keygen
```

2. 公開鍵（`id_rsa.pub` の内容）を GitHub の「Settings → SSH and GPG keys」に登録する。
3. リモート URL を SSH 形式に切り替える：

```bash
git remote set-url origin git@github.com:hoso-jpn/<リポジトリ名>.git
```

### ディレクトリ構造の確認コマンド

#### `ls`：ファイル一覧の表示

```bash
ls          # 通常表示
ls -a       # 隠しファイル（.git など）も含めて表示
ls -la      # 権限・所有者・サイズ・更新日時を詳細表示
```

> `.git` フォルダが確認できれば、Git 管理が正常に開始されている証拠です。

#### `tree`：ディレクトリ構造を樹形図で表示

```bash
tree            # 全階層を表示
tree -L 2       # 2 階層目まで表示（大規模プロジェクトに便利）
tree -I ".venv" # 特定フォルダを除外して表示
```

> Python プロジェクトでは `.venv` 内にライブラリが大量にあるため、`-I ".venv"` の使用を推奨します。

#### `git status`：Git 視点での状態確認

```bash
git status
```

クローン直後に `nothing to commit, working tree clean` と表示されれば、ローカルとリモートの状態が完全に一致しています。
